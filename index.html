Class & Object

The blueprint and the instance — the foundation of every OOP program.

Class blueprint

A user-defined data type that groups data members (attributes) and member functions (methods) under one name. It acts as a template — no memory is allocated until an object is created.

Object instance

A real-world entity created from a class. When an object is instantiated, memory is allocated for its data members. Multiple objects can be created from the same class, each with their own state.

Access Specifiers

public: Accessible everywhere.
private: Only within the class (default).
protected: Class + derived classes.

class_object.cpp 
// ── Class Definition ──────────────────────────────────────
class Car {
private:                          // hidden from outside
    string brand;
    int    speed;

public:                           // accessible anywhere
    // Setter
    void setBrand(string b) { brand = b; }
    void setSpeed(int s)    { speed = s; }

    // Getter
    string getBrand() { return brand; }
    int    getSpeed() { return speed; }

    void display() {
        cout << brand << " runs at " << speed << " km/h\n";
    }
};

// ── Object Creation ───────────────────────────────────────
int main() {
    Car c1;                       // object on stack
    c1.setBrand("Toyota");
    c1.setSpeed(120);
    c1.display();                // Toyota runs at 120 km/h

    Car* c2 = new Car();          // object on heap
    c2->setBrand("BMW");
    delete c2;                    // free heap memory
}

constructors.cpp 
class Student {
public:
    string name;
    int age;

    // ① Default Constructor
    Student() {
        name = "Unknown";
        age  = 0;
        cout << "Default constructor called\n";
    }

    // ② Parameterised Constructor
    Student(string n, int a) : name(n), age(a) {   // initialiser list
        cout << "Parameterised constructor called\n";
    }

    // ③ Copy Constructor
    Student(const Student& s) {
        name = s.name;
        age  = s.age;
        cout << "Copy constructor called\n";
    }
};

int main() {
    Student s1;                        // Default
    Student s2("Alice", 20);           // Parameterised
    Student s3(s2);                    // Copy
    Student s4 = s2;                   // Also calls Copy constructor
}
Constructor Overloading
✅
Constructor Overloading Defining multiple constructors with different parameter lists in the same class. The compiler selects the right one based on the arguments passed.

⚡ Quick Revision — Constructors

  • Same name as class, no return type, auto-called at object creation
  • Default: no params — sets members to default values
  • Parameterised: accepts args — use initialiser list for efficiency
  • Copy constructor: takes const ClassName& — called on copy/pass-by-value
  • Constructors can be overloaded but NOT inherited or virtual
  • If you define any constructor, the compiler no longer provides a default one

Shallow Copy vs Deep Copy

How objects are duplicated — critical when your class manages dynamic (heap) memory.

💡
Why Does This Matter? When a class contains a raw pointer to heap memory, the default compiler-generated copy constructor only copies the pointer address — not the actual data. This is a Shallow Copy. It causes double-deletion crashes and data corruption. A Deep Copy allocates brand-new memory and duplicates the actual data.
🪞 Shallow Copy compiler default

Copies only the pointer address, not the data it points to. Both objects share the same heap block.

⚠️ When one object is destroyed, delete frees shared memory. The other now holds a dangling pointer → undefined behaviour + crash!
🔒 Deep Copy must write manually

Allocates brand-new memory and copies the actual data into it. Original and copy are completely independent.

✅ Each object owns its own memory. Both destructors run safely with no double-free or data corruption.

shallow_vs_deep_copy.cpp 
class Array {
private:
    int*  data;
    int   size;
public:
    Array(int s) : size(s) {
        data = new int[size];
        for (int i = 0; i < size; i++) data[i] = i * 10;
    }

    // ❌ Shallow (compiler default — DO NOT rely on this for pointer classes)
    // Array(const Array& o) { size=o.size; data=o.data; }  ← copies address only!

    // ✅ Deep Copy Constructor
    Array(const Array& o) : size(o.size) {
        data = new int[size];             // allocate fresh memory
        for (int i = 0; i < size; i++)
            data[i] = o.data[i];           // copy actual values
    }

    // ✅ Deep Copy Assignment Operator
    Array& operator=(const Array& o) {
        if (this == &o) return *this;     // self-assignment guard
        delete[] data;                     // free OLD memory first!
        size = o.size;
        data = new int[size];
        for (int i = 0; i < size; i++) data[i] = o.data[i];
        return *this;
    }

    ~Array() { delete[] data; }         // destructor — frees its OWN memory

    void modify(int i, int v) { data[i] = v; }
    void print() { for(int i=0;i<size;i++) cout<<data[i]<<" "; cout<<"\n"; }
};

int main() {
    Array a(3);           // a → [0, 10, 20]
    Array b(a);            // Deep copy → b gets its OWN [0, 10, 20]

    b.modify(0, 99);      // modify b only
    a.print();             // 0 10 20  ← a is UNAFFECTED ✅
    b.print();             // 99 10 20
}                          // both destructors run safely — no double-free!

Rule of Three (C++03) / Rule of Five (C++11) If your class needs a custom Destructor, it almost certainly also needs a custom Copy Constructor and Copy Assignment Operator (Rule of Three). In C++11 add Move Constructor and Move Assignment Operator too — that is the Rule of Five.

Destructor

Automatically called when an object goes out of scope or is deleted.

Syntax

Same name as class with a tilde ~ prefix.
No parameters, no return type.
Only ONE destructor per class.
Cannot be overloaded.

When is it Called?

• Stack object → exits its scope block
• Heap object → delete is called
• Static object → end of program
• Called in reverse order of construction

Why Important?

Essential for releasing dynamic memory, closing files, releasing network connections, and preventing memory leaks.

destructor.cpp 
class Buffer {
private:
    int* data;
public:
    Buffer(int size) {
        data = new int[size];           // allocate on heap
        cout << "Buffer created\n";
    }

    ~Buffer() {                         // Destructor
        delete[] data;                  // free memory ← crucial!
        cout << "Buffer destroyed\n";
    }
};

int main() {
    {
        Buffer b(100);                  // constructor called
    }                                   // destructor called here (scope ends)

    Buffer* p = new Buffer(50);
    delete p;                           // destructor called explicitly
}

Encapsulation

Wrapping data and functions into a single unit and restricting direct access.

📦
Core Idea Data is declared private and can only be accessed/modified through public getter/setter functions. This prevents accidental corruption of internal state.

🛡️ Data Hiding

Private members are invisible outside the class — only the class knows its own internals.

🔧 Controlled Access

Getters and setters act as controlled gates — you can add validation inside them.

🔄 Abstraction Support

Users interact with a clean interface, unaware of internal complexity.

encapsulation.cpp 
class BankAccount {
private:
    double balance;                    // hidden — can't touch directly

public:
    BankAccount(double b) : balance(b) {}

    void deposit(double amt) {
        if (amt > 0) balance += amt;    // validated setter
    }
    void withdraw(double amt) {
        if (amt <= balance) balance -= amt;
        else cout << "Insufficient funds!\n";
    }
    double getBalance() const { return balance; }
};

int main() {
    BankAccount acc(1000.0);
    acc.deposit(500);
    acc.withdraw(200);
    // acc.balance = -999;  ← COMPILE ERROR — private!
    cout << acc.getBalance();       // 1300
}

class Animal {
public: void eat() { cout << "Eating\n"; }
};
class Dog : public Animal {          // single base
public: void bark() { cout << "Woof!\n"; }
};

// ── ② Multiple Inheritance ────────────────────────────────
class Flyable { public: void fly()  { cout << "Flying\n"; } };
class Swimmable { public: void swim() { cout << "Swimming\n"; } };
class Duck : public Flyable, public Swimmable {};

// ── ③ Multilevel Inheritance ──────────────────────────────
class Puppy : public Dog {           // Dog inherits Animal, Puppy inherits Dog
public: void play() { cout << "Playing!\n"; }
};

class Cat : public Animal {};       // multiple classes from same base
class Bird : public Animal {};

// ── ⑤ Diamond Problem & Virtual Inheritance ──────────────
class Base  { public: int x = 10; };
class Left  : virtual public Base {}; // virtual keyword saves us!
class Right : virtual public Base {};
class Bottom : public Left, public Right {};
// Without virtual: Bottom has TWO copies of Base::x → ambiguity error
// With    virtual: Only ONE shared copy of Base::x  → resolved!

int main() {
    Bottom b;
    cout << b.x;  // 10 — unambiguous because of virtual inheritance
}
⚠️
The Diamond Problem When two parent classes share the same grandparent, the grandchild gets duplicate copies of grandparent members. Solution: use virtual inheritance so only ONE shared copy exists.
🔷 Diamond Problem — In Depth
💡
Core Idea The Diamond Problem occurs when class D inherits from classes B and C, and both B and C inherit from the same base class A. The inheritance hierarchy forms a diamond shape. Without virtual, class D ends up with two separate copies of A's members, and the compiler cannot decide which one to use — this is called ambiguity.

❌ The Problem

D obj;
obj.show();  // COMPILE ERROR

The compiler is confused — two inheritance paths both lead to a copy of A::show():
• Path 1: D → B → A
• Path 2: D → C → A

Result: 2 copies of A exist in D → ambiguity!

✅ The Solution

Add virtual when B and C inherit from A:

class B : virtual public A {};
class C : virtual public A {};

virtual tells the compiler: "share one single copy of A across the whole hierarchy."

✔️ Only ONE copy of A in D — ambiguity resolved!

⚡ Workaround (No virtual)

Before virtual inheritance, you could force a specific path using scope resolution:

obj.B::show();  ← uses B's path
obj.C::show();  ← uses C's path

⚠️ Verbose and fragile. virtual is always the correct fix.

❌ Problem Code — Without virtual
diamond_problem.cpp 
class A {
public:
    int x = 10;
    void show() { cout << "Class A, x = " << x << endl; }
};

class B : public A {};   // B gets its own copy of A
class C : public A {};   // C gets its own copy of A

class D : public B, public C {};  // D now has TWO copies of A!

int main() {
    D obj;
    obj.show();      // ❌ COMPILE ERROR: ambiguous — B::A::show() or C::A::show()?
    obj.x = 5;      // ❌ COMPILE ERROR: which x? B's copy or C's copy?

    // Workaround with scope resolution (ugly, not recommended):
    obj.B::show();  // forces B's path — compiles but is verbose
    obj.C::show();  // forces C's path
}
✅ Solution Code — Virtual Inheritance
diamond_solution.cpp 
class A {
public:
    int x = 10;
    void show() { cout << "Class A, x = " << x << endl; }
};

// virtual keyword → B and C share ONE instance of A instead of getting their own
class B : virtual public A {};
class C : virtual public A {};

class D : public B, public C {};

int main() {
    D obj;
    obj.show();          // ✅ "Class A, x = 10" — unambiguous
    obj.x = 99;          // ✅ only ONE x exists — no conflict
    cout << obj.x;         // 99

    // D's constructor is responsible for constructing the virtual base A:
    // Constructor order: A → B → C → D   (A is always first, built once)
}
🔢 Constructor Call Order with Virtual Inheritance
⚠️
Important Rule With virtual inheritance, the most-derived class (D) is responsible for constructing the virtual base class A directly — not B or C. This prevents A from being constructed multiple times.
  • 1
    A's constructor is called first (the shared virtual base) — regardless of what B and C say.
  • 2
    B's constructor runs next — B's call to A's constructor is silently ignored since A is already built.
  • 3
    C's constructor runs — again, its call to A is skipped.
  • 4
    D's constructor runs last. Destructor order is exactly reversed: D → C → B → A.
constructor_order.cpp 
class A { public: A() { cout << "A constructor\n"; } };
class B : virtual public A { public: B() { cout << "B constructor\n"; } };
class C : virtual public A { public: C() { cout << "C constructor\n"; } };
class D : public B, public C  { public: D() { cout << "D constructor\n"; } };

int main() {
    D obj;
    // Output:
    // A constructor   ← virtual base, built FIRST and ONCE by D
    // B constructor
    // C constructor
    // D constructor
}

🎯 Exam Q&A — Diamond Problem
Q1 — Definition
"What is the Diamond Problem in C++?"
When class D inherits from B and C, and both B and C inherit from A, D gets two copies of A's members. This creates ambiguity — the compiler cannot determine which copy to use.
Q2 — Solution
"How do you solve the Diamond Problem?"
Use virtual inheritance when B and C inherit from A: class B : virtual public A. This ensures only one shared copy of A exists throughout the hierarchy.
Q3 — Constructor Order
"What is the constructor call order with virtual inheritance?"
With virtual inheritance, the most-derived class (D) constructs the virtual base A first — directly. Order: A → B → C → D. Destructors run in reverse: D → C → B → A.
Q4 — Without virtual
"What happens if you don't use virtual?"
D will contain two separate subobjects of A. Any attempt to access A's members directly through D results in a compile-time ambiguity error. You must use scope resolution (B:: or C::) which is verbose and error-prone.
Q5 — Real World
"Give a real-world analogy of the Diamond Problem."
A FlyingFish inherits from Bird and Fish, both of which inherit from Animal. Without virtual, FlyingFish has two copies of Animal — which eat() do you call? Virtual inheritance ensures there is only one Animal base shared by Bird and Fish.
Q6 — vptr
"What overhead does virtual inheritance add?"
Virtual inheritance adds a vptr (virtual base pointer) in intermediate classes (B, C) that points to the shared A subobject. This adds a small memory and indirection overhead, but the benefit — eliminating ambiguity — far outweighs it.

⚡ Quick Revision — Diamond Problem

  • Diamond = A (base) → B, C (middle) → D (bottom) inheritance
  • Without virtual: D holds 2 copies of A → compile-time ambiguity error
  • Fix: add virtual in B and C when they inherit from A
  • With virtual: D holds 1 shared copy of A → no ambiguity
  • Constructor order (virtual): A → B → C → D (D builds A directly, once)
  • Workaround without virtual: scope resolution obj.B::show() — but messy
  • Virtual inheritance adds a vptr for indirection — tiny overhead, worth it

Polymorphism

"Many forms" — same function name behaves differently in different contexts.

Compile-Time Polymorphism

Also called Static or Early Binding. Resolved by the compiler before the program runs.

Function Overloading Operator Overloading

Runtime Polymorphism

Also called Dynamic or Late Binding. Resolved at runtime through virtual functions and base class pointers.

Function Overriding Virtual Functions

⚡ Quick Revision — Polymorphism

  • Compile-time (Static): Function overloading, Operator overloading — resolved by compiler
  • Runtime (Dynamic): Function overriding via virtual functions — resolved at runtime
  • Runtime polymorphism requires a base class pointer/reference pointing to a derived object
  • Without virtual → static binding (always calls base version)
  • With virtual → dynamic binding (calls correct derived version at runtime)

Function Overloading

Compile-time polymorphism — same name, different parameter signatures.

📝
Rules for Overloading Functions must differ in: number of parameters, types of parameters, or order of parameters. Return type alone is NOT sufficient to overload.
function_overloading.cpp 
class Calculator {
public:
    // Different number of params
    int add(int a, int b)          { return a + b; }
    int add(int a, int b, int c)   { return a + b + c; }

    // Different param types
    double add(double a, double b) { return a + b; }

    // Different order
    void show(int i, double d)     { cout << i << " " << d; }
    void show(double d, int i)     { cout << d << " " << i; }
};

int main() {
    Calculator c;
    cout << c.add(3, 4);          // calls int version → 7
    cout << c.add(3, 4, 5);       // 3-param version → 12
    cout << c.add(1.5, 2.5);      // double version → 4.0
}

⚠️
Without virtual → Static Binding Base class pointer always calls base version, even if pointing to derived object. Use virtual keyword in base to enable dynamic dispatch.
virtual_function.cpp 
class Shape {
public:
    virtual void draw() {            // virtual → runtime dispatch
        cout << "Drawing Shape\n";
    }
    virtual ~Shape() {}             // always make destructor virtual!
};

class Circle : public Shape {
public:
    void draw() override {           // override keyword (C++11)
        cout << "Drawing Circle ○\n";
    }
};

class Rectangle : public Shape {
public:
    void draw() override {
        cout << "Drawing Rectangle □\n";
    }
};

int main() {
    Shape* shapes[] = { new Circle(), new Rectangle() };

    for (auto* s : shapes)
        s->draw();                    // resolved at RUNTIME
    // Output:
    // Drawing Circle ○
    // Drawing Rectangle □
}

// ── Pure Virtual (Abstract Class) ─────────────────────────
class AbstractShape {
public:
    virtual double area() = 0;     // pure virtual — MUST override
};
// AbstractShape s;  ← ERROR — cannot instantiate abstract class

⚡ Quick Revision — Virtual Functions & Overriding

  • Overriding: same signature in derived class — must use virtual in base
  • Use override keyword in derived (C++11) — compiler checks signature match
  • Without virtual: base pointer always calls base version (static binding)
  • With virtual: correct derived version is called at runtime (dynamic dispatch via vtable)
  • Pure virtual: virtual void fn() = 0; — makes class abstract, cannot instantiate
  • Always make destructor virtual in base class — prevents memory leaks with polymorphism

Operator Overloading & Friend Functions

Give custom meaning to operators for user-defined types.

💡
Syntax: returnType operator@(params)
where @ is any overloadable operator like +, -, ==, <<, etc.
👥
Friend Function A friend function is not a member of a class but can access its private and protected members. Essential for overloading stream operators (<<, >>) and binary operators needing LHS access.
operator_overloading.cpp 
class Complex {
private:
    double real, imag;
public:
    Complex(double r = 0, double i = 0) : real(r), imag(i) {}

    // ── Member function overloading (+) ───────────────────
    Complex operator+(const Complex& other) const {
        return Complex(real + other.real, imag + other.imag);
    }

    // ── Member function overloading (==) ──────────────────
    bool operator==(const Complex& o) const {
        return (real == o.real) && (imag == o.imag);
    }

    // ── Unary operator overloading (-) ────────────────────
    Complex operator-() const {
        return Complex(-real, -imag);
    }

    // ── Friend function to overload << ────────────────────
    friend ostream& operator<<(ostream& out, const Complex& c) {
        out << c.real;                 // accesses private members!
        if (c.imag >= 0) out << "+";
        out << c.imag << "i";
        return out;                    // allows chaining cout << a << b
    }

    // ── Friend for binary ops where LHS isn't the class ───
    friend Complex operator*(double scalar, const Complex& c) {
        return Complex(scalar * c.real, scalar * c.imag);
    }
};

int main() {
    Complex a(3, 4), b(1, 2);
    Complex c = a + b;
    cout << c;                        // 4+6i  (uses friend <<)

    Complex d = 2.0 * a;              // uses friend * (double on LHS)
    cout << d;                        // 6+8i

    cout << (a == b ? "Equal" : "Not Equal"); // Not Equal
}

Best Practices Don't change the fundamental meaning of operators (e.g., don't make + behave like subtraction). Always return the correct type to allow chaining. Make the destructor virtual in base classes.
